### WP2: Intra-file cross-function data flow tracing (DEPTH MOAT)

**Problem:** Current `_FlowTracer` in ast_analysis.py traces within function bodies only. `def process(data): return model.predict(data)` called from `def handle(): result = process(input)` — the flow from model.predict through process to handle's return is not traced. This is documented as a known gap in docs/audit/ast_and_compliance_audit.md:208.

**Scope:**
- Build per-file function call graph (visitor that maps function_name → functions_it_calls)
- For each function, record what it returns (extend existing `_var_origins` and `_value_origin()`)
- When function A calls function B and uses the result, propagate B's AI origins to A's variable
- Wire into Article 14 (oversight) and Article 12 (logging) scoring — the two articles that benefit most from cross-function visibility

**What this enables that competitors cannot do:**
- Detect "AI model called in service function, decision made in handler function" (most common real-world pattern)
- Detect "oversight check exists in middleware but AI call is in a different function in the same file"
- Score compliance gaps more accurately because the analysis sees the full intra-file data flow

**What this does NOT include (explicitly deferred):**
- Cross-file tracing (import resolution, multi-file call graphs) — deferred per STRATEGY.md
- Multi-language cross-function (tree-sitter JS/TS doesn't have FlowTracer equivalent)
- Protected-class semantic tracing (requires type inference that stdlib ast cannot provide)

**Acceptance criteria:**
- New tests covering cross-function flow scenarios
- Existing 438 tests still pass (no regressions)
- `regula gap` produces higher-quality Article 14 and Article 12 scores when oversight/logging is in a different function than the AI call
- Performance: no more than 2x slowdown on current test suite

**Effort:** 2-3 days

### WP5: Benchmark 5 real AI projects (DATA MOAT SEED)

**Problem:** 0 real-world validation. False positive rate unknown. No calibration data.

**Scope:**
- Clone 5 high-profile OSS AI projects: instructor, pydantic-ai, langchain, scikit-learn, openai-python
- Run `regula check` on each
- Manually label every finding as TP (true positive), FP (false positive), or DISPUTED
- Calculate precision and recall per risk tier
- Store labelled results in `benchmarks/` directory as the beginning of the calibration dataset
- If precision < 80% on any tier, tune the responsible patterns before publishing

**Why these 5:**
- instructor: small, focused, high-risk patterns (structured output from LLMs)
- pydantic-ai: moderate size, agent patterns
- langchain: large, complex, many AI patterns
- scikit-learn: classical ML, training patterns, bias-relevant
- openai-python: API client, credential patterns

**Acceptance criteria:**
- Labelled findings for all 5 projects stored in benchmarks/
- Precision ≥ 80% per tier (or patterns tuned until achieved)
- Recall measured (what did Regula miss?)
- Summary statistics ready for the article (WP7)

**Effort:** 2 days

Open-core tier split is **deferred** until 500+ weekly installs. Data shows every successful open-core company needed massive free adoption first (Semgrep: 2M+ users, Sentry: years of free adoption, GitLab: 100K+ users before enterprise revenue).

**Immediate revenue path:** Consulting accelerator. Use Regula as intake/demo tool for EU AI Act compliance consulting. "I'll scan your codebase and produce a gap assessment in a day, not a quarter." EUR 5,000-15,000 per engagement. Can start immediately, no users required.

**Paid tier boundary (when ready):** Organisational features (dashboards, audit trails, team management, compliance packages), NOT technical depth. Gating AST analysis would kill the core differentiator.
